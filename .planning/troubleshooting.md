    - **현상**: 폴더를 모니터링에서 해제했다가 다시 등록하면, `search` 결과에 동일한 파일이 2개 이상 중복되어 나타남.
    - **원인 분석**:
        1. **벡터 DB 삭제 미구현**: `remove_from_monitoring` 수행 시 RDB(SQLite)에서는 파일 정보가 삭제되지만, 벡터 DB(FAISS)에서는 해당 파일의 임베딩 벡터가 삭제되지 않고 그대로 남음.
        2. **재인덱싱 시 추가 적재**: 폴더를 재등록하면 새로운 임베딩 벡터가 추가되어 중복 발생.
        3. **검색 로직의 중복 허용**: `search` 메서드가 벡터 DB 결과를 그대로 반환.
    - **조치 내용**:
        1. **검색 중복 제거**: `SemanticIndexer.search` 메서드에 `file_path` 기준 중복 제거 로직 추가.
        2. **Vector DB 구조 개선**: `IndexIDMap` 및 `IndexFlatIP`(Cosine Similarity) 도입으로 벡터 ID 관리 체계 구축.
        3. **삭제 로직 구현**: 파일/폴더 삭제 및 수정 시 RDB와 연동하여 Vector DB의 해당 벡터 ID를 조회하고 선행 삭제하도록 로직 개선 (`scanner.py`, `indexer.py`).
        4. **스키마 변경**: `file_vectors` 테이블을 추가하여 파일과 벡터 ID 간의 매핑 정보 관리.
    - **결과**: 파일 수정, 삭제, 폴더 재등록 시에도 벡터 DB에 중복 데이터가 남지 않으며 검색 결과가 정확해짐. (검증 스크립트 통과)
